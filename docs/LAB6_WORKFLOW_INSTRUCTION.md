## 2 Теоретические сведения

### 2.1 Переходная система

Динамика параметризованного информационного объекта описывается **переходной системой**.
Состояние — снимок конфигурации `⟨сеть, контекст, шаг, режим⟩`. Изменение состояния
задаётся **правилами переходов**.

**Правило перехода** — кортеж `(имя, гвард, эффект, приоритет, событие)`:

| Элемент | Описание |
|---|---|
| гвард (guard) | Формула-условие применимости; правило применимо, если гвард истинен (TRUE) |
| эффект (effect) | Последовательность операций над фактами: `add` (добавить) / `retract` (снять) |
| приоритет (priority) | При нескольких применимых правилах выбирается правило наибольшего приоритета |
| событие (on_event) | Если задано, правило срабатывает только при поступлении одноимённого события |

### 2.2 Переходный и устойчивый режимы

Каждое состояние классифицируется по режиму:

| Режим | Условие |
|---|---|
| **Переходный** (transient) | существует применимое правило **или** нарушены инварианты |
| **Устойчивый** (stable) | достигнута неподвижная точка (нет применимых правил) **и** инварианты выполнены |

Прогон до неподвижной точки последовательно применяет автономные правила (без событий),
пока состояние не станет устойчивым либо не будет достигнут лимит шагов. **Время
сходимости** — число шагов до выхода в устойчивый режим.

### 2.3 Инварианты

**Инвариант** — формула (или набор формул), которая должна выполняться в устойчивом
состоянии. Инварианты выражают денотационную согласованность объекта. В сценарии
документооборота инвариант — «документ всегда находится в одном из известных статусов»:

```
Or( status(doc, draft), status(doc, review), status(doc, approved),
    status(doc, published), status(doc, rejected) )
```

После каждого шага система проверяет инварианты; нарушение отражается в трассе и
переводит состояние в переходный режим.

### 2.4 Трасса переходов

**Трасса** — структурированная последовательность шагов. Каждый шаг фиксирует:
номер, сработавшее правило, событие, добавленные/снятые факты, статус инвариантов и
режим. Трасса обеспечивает **объяснимость**: однозначно показывает происхождение
текущего состояния.

### 2.5 Формальная верификация

**Ограниченный model-checker** (`check_model`) исчерпывающе обходит все достижимые
состояния, применяя из каждого все применимые правила, и проверяет инварианты. Если
инвариант нарушается, возвращается **кратчайший контрпример** — путь правил до
состояния-нарушителя. Дополнительно система специфицируется на TLA+ (`specs/`).

### 2.6 Экспериментальные метрики

По трассе вычисляются метрики:

| Метрика | Описание |
|---|---|
| Convergence Time | число шагов до устойчивого режима |
| Constraint Satisfaction Rate (CSR) | доля шагов с выполненными инвариантами |
| Invariant Violations | число шагов с нарушением инвариантов |

### 2.7 Декларативный DSL

Модель (сеть + правила + инварианты) описывается единым документом JSON/YAML и
загружается функциями `ctmsn.io.load_model` / `dump_model` (round-trip).

## 3 Требования к оборудованию и ПО

- Python 3.9+ (ядро — без внешних зависимостей).

## 4 Порядок выполнения

### Часть A. Переходы в веб-интерфейсе

**Шаг 1. Загрузка сценария.** Запустите `make dev`, войдите в систему, создайте новый
workspace. В панели сценария выберите `doc_workflow` и нажмите «Загрузить». На графе
отобразятся документ, статусы и рецензенты; документ имеет статус `draft`.

**Шаг 2. Создание инварианта.** В панели «Конструктор формул» создайте формулу с именем
`consistency` типа `Or`, добавив пять атомов `status(doc, s)` для каждого статуса
(`draft`, `review`, `approved`, `published`, `rejected`).

**Шаг 3. Построение правил переходов.** В панели «Переходы (режимы)» нажмите
«+ Правило» и создайте три автономных правила:

| Имя | Гвард (FactAtom) | Эффект |
|---|---|---|
| `submit` | `status(doc, draft)` | retract `status(doc, draft)`; add `status(doc, review)` |
| `approve` | `status(doc, review)` | retract `status(doc, review)`; add `status(doc, approved)` |
| `publish` | `status(doc, approved)` | retract `status(doc, approved)`; add `status(doc, published)` |

**Шаг 4. Запуск переходов.** В разделе «Инварианты» отметьте формулу `consistency`.
Нажмите «Выполнить переходы». Убедитесь, что:
- итог — «Устойчивый режим за 3 шаг(ов)»;
- трасса содержит шаги `submit → approve → publish`;
- на каждом шаге инварианты выполнены (нет пометки «инвариант!»).

**Шаг 5. Пошаговая подсветка.** Кликните по шагу трассы — затронутые узлы графа
подсветятся. Проследите перемещение документа по статусам.

### Часть B. Конвейер из командной строки

**Шаг 6. Полный прогон сценария.** Выполните демонстрацию:

```bash
python3 src/ctmsn/examples/doc_workflow_demo.py
```

Зафиксируйте в отчёте: результат форсинга до переходов (цель ложна), трассу переходов,
результат форсинга после переходов (`force` подбирает рецензента `alice`), итог
верификации, метрики и результат round-trip DSL.

**Шаг 7. Форсинг до и после переходов.** Объясните, почему до переходов
`forces = FALSE` (документ — черновик, цель содержит `status(doc, published)`), а после
переходов `force` находит расширение контекста `reviewer = alice`.

**Шаг 8. Формальная верификация.** В интерактивном Python:

```python
from ctmsn.scenarios.doc_workflow.model import build_network
from ctmsn.scenarios.doc_workflow.transition import build_rules, build_invariants
from ctmsn.transition import check_model

net = build_network()
res = check_model(net, build_rules(net), build_invariants(net))
print(res.invariant_holds, res.states_explored, res.counterexample)
```

Убедитесь, что инвариант держится во всех достижимых состояниях (контрпример
отсутствует).

**Шаг 9. Метрики.** Выполните:

```python
from ctmsn.experiment import run_case
from ctmsn.scenarios.doc_workflow.experiment import build_case
print(run_case(build_case()).metrics)
```

Зафиксируйте Convergence Time и CSR.

**Шаг 10. DSL round-trip.** Выполните:

```python
from ctmsn.io import dumps_model_json, loads_model_json
from ctmsn.scenarios.doc_workflow.model_doc import build_model

doc = dumps_model_json(build_model())
print(doc[:300])                        # фрагмент JSON-описания модели
restored = loads_model_json(doc)        # загрузка обратно
print(len(restored.rules), len(restored.invariants.items))
```

Сохраните JSON-описание модели как артефакт.
